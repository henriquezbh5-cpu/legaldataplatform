LegalDataPlatform

Manual de uso sencillo · Plataforma ETL/ELT para datos legales y comerciales en Python + AWS + PostgreSQL

Para abrir el proyecto en tu navegador:
1) Levanta el API → make api (o python -m uvicorn src.api.main:app --port 8000)
2) Abre http://127.0.0.1:8000/ · Swagger en http://127.0.0.1:8000/docs

¿Qué es esta plataforma?

LegalDataPlatform es una plataforma ETL/ELT que ingesta datos legales y comerciales desde múltiples fuentes (SEC EDGAR, GLEIF, CSV, APIs REST), los normaliza, valida con reglas de calidad y los carga en PostgreSQL particionado. Sigue arquitectura medallion Bronze → Silver → Gold y está pensada para correr local con Docker o en AWS (S3 + Lambda + Glue + ECS).

1. Desarrollo de pipelines ETL/ELT en Python y AWS

Requisito del puesto: Diseñar y mantener pipelines ETL/ELT escalables en Python y AWS para la ingesta de datos de diversas fuentes legales y comerciales.
Pipelines orquestados con Prefect, escritos en Python 3.11+. Cargas concurrentes con asyncio, transformaciones con Polars y Pydantic, despliegue en AWS vía Lambda + Glue + ECS. Cuatro flujos de ingesta listos: SEC EDGAR, GLEIF, CSV legal y comercial.

Cómo correr un pipeline (paso a paso)

Paso 1. Levanta el stack local: make up (arranca Postgres, MinIO, Prefect, Grafana en Docker)
Paso 2. Aplica migraciones: make migrate
Paso 3. Corre un flujo. Tienes 4 disponibles:
  • make pipeline — ingesta de documentos legales desde CSV
  • make sec-edgar — ingesta desde la API real de SEC EDGAR
  • make gleif — ingesta de identificadores LEI desde GLEIF
  • make commercial — ingesta de transacciones comerciales
Paso 4. Observa el run en Prefect UI → http://localhost:4200
Paso 5. Verifica el archivo Parquet en MinIO → http://localhost:9001 (bucket bronze)

Flujo end-to-end (ejemplo: documentos legales)

  1. Extract — CSVExtractor lee en chunks de 5,000 filas con asyncio (no carga todo en memoria)
  2. Bronze — S3ParquetLoader escribe Parquet particionado por fecha en S3/MinIO
  3. Normalize — Pydantic valida cada registro contra LegalDocumentSchema
  4. Quality gate — el motor de reglas YAML aplica validaciones declarativas
  5. Load — PostgresBulkLoader usa COPY + UPSERT hacia tabla particionada
  6. Gold — refresca Materialized Views CONCURRENTLY (sin bloquear lecturas)

En AWS

  • S3 PUT → Lambda (src/aws/lambda_handlers/s3_event_ingest.py) encola un mensaje en SQS
  • SQS → Lambda (sqs_pipeline_trigger.py) dispara el pipeline en ECS
  • AWS Glue (src/aws/glue_jobs/legal_doc_normalize.py) escala normalización con Spark
  • Despliegue documentado en docs/AWS_DEPLOYMENT.md e infra en infra/

2. Optimización de PostgreSQL

Requisito del puesto: Gestionar y optimizar bases de datos PostgreSQL, garantizando alto rendimiento para datos relacionales complejos.
Particionado por rango mensual, índices BRIN/GIN/expression/partial/covering, Materialized Views con refresh CONCURRENTLY, suite de benchmarks reproducibles y migraciones versionadas con Alembic. Documentado en docs/POSTGRESQL_OPTIMIZATION.md.

Optimizaciones aplicadas (lo que verás en producción)

  • Particionado RANGE mensual en legal_documents y transactions — el planner ignora particiones irrelevantes y VACUUM/REINDEX trabajan por partición
  • Índice BRIN en transaction_date → ~10KB/millón de filas vs ~30MB de un BTREE
  • Covering index con INCLUDE para queries de "suma últimos 90 días" → index-only scans
  • Índices parciales para contratos activos (filtra WHERE status IN ('ACTIVE','PENDING'))
  • Índices GIN sobre columnas JSONB (tags, metadata) para búsquedas por tag/atributo
  • Índice de expresión lower(legal_name) para búsquedas case-insensitive
  • Materialized Views precomputan revenue mensual por contraparte; refresh CONCURRENTLY no bloquea lectores
  • UPSERT vía COPY + tabla staging en lugar de INSERT fila por fila → 10–50× más rápido

Cómo medir performance

Paso 1. Ejecuta los benchmarks: make benchmark
Paso 2. Captura evidencia con make evidence → genera docs/evidence/benchmarks.md
Paso 3. Revisa Grafana en http://localhost:3000 para ver latencia de queries en tiempo real

3. Calidad y normalización de datos (Fuente de verdad)

Requisito del puesto: Implementar procesos rigurosos de validación y limpieza que mantengan los estándares de "Fuente de verdad".
Doble capa de validación: (1) Pydantic para tipado y normalización, (2) reglas YAML declarativas con severidades error/warning. Registros que fallan reglas error bloquean el pipeline y van a un bucket de cuarentena en S3 con el motivo. Integración opcional con Great Expectations.

Cómo se aplica el principio "Fuente de verdad"

  1. Cada registro entra a Bronze tal cual (auditoría completa)
  2. Pydantic intenta normalizarlo. Si falla → va a Quarantine con el error
  3. El motor de reglas YAML aplica controles. Reglas error → bloquean el flow vía assert_no_errors()
  4. Reglas warning → emiten métricas Prometheus pero no bloquean
  5. Solo registros que pasan ambas capas llegan a Silver (la "fuente de verdad")
  6. Cada registro lleva un source_hash SHA-256 para idempotencia y trazabilidad

Cómo agregar una nueva regla (sin tocar código)

Paso 1. Edita src/data_quality/rules/legal_documents.yaml
Paso 2. Agrega un bloque como: 
- name: title_min_length
  type: min_length
  column: title
  params: { min: 5 }
  severity: warning
Paso 3. Re-ejecuta el pipeline. La regla se aplicará automáticamente.
