01 · Contexto

El problema que resuelve

Una empresa consume datos de múltiples proveedores externos: APIs regulatorias, sistemas ERP, archivos CSV de terceros, bases de datos transaccionales. Cada fuente trae su propio formato, sus errores, sus duplicados y sus cambios.

Sin una plataforma central, el equipo de negocio termina tomando decisiones sobre datos inconsistentes, desactualizados o directamente erróneos.

Fuente 1

APIs REST

Registros públicos, regulaciones

Fuente 2

Archivos planos

CSV de clientes y contrapartes

Fuente 3

Bases externas

ERPs, bancos, sistemas legacy

Fuente 4

Data lakes

Archivos masivos en S3

Objetivo: un sistema que ingiera, normalice, valide y consolide todo en una única "fuente de verdad" auditable, rápida y confiable.
02 · Analogía central

Piénsalo como un centro de distribución moderno

Imagina el hub logístico de una empresa como Amazon o DHL. Llegan contenedores de todo el mundo con productos de miles de proveedores. Antes de que cualquier producto termine en manos de un cliente, pasa por un proceso estandarizado.

Muelle de recepción → Control de calidad → Zona de clasificación ↓ Almacén principal → Preparación de pedido → Despacho al cliente

Nuestra plataforma funciona exactamente igual, pero con datos en vez de productos:

Ingesta (extractors) → Validación (DQ) → Normalización ↓ Base de datos (SoT) → Agregación (gold) → Consumo (BI/ML/APIs)
Principio rector: cada etapa tiene una responsabilidad única y acotada. Ningún componente hace "de todo". Esto permite escalar, debuggear y reemplazar piezas sin romper el resto.
03 · Arquitectura Medallion

Cuatro zonas con propósitos distintos

El centro de distribución no tiene un solo almacén: tiene zonas especializadas. Lo mismo aplica a los datos.

Raw

Bronze

Zona de recepción. Los datos llegan tal cual, sin tocar. Inmutable. Sirve como respaldo histórico.

Clean

Silver

Datos limpios, validados, normalizados. Listos para consumo operativo.

Business

Gold

Agregaciones y vistas de negocio. El "producto terminado" que consumen dashboards y reportes.

Rejected

Quarantine

Datos que fallaron validación. Se aíslan con el motivo del rechazo para triaje posterior.

¿Por qué conservar Bronze si los datos ya están limpios en Silver? Porque si mañana cambiamos una regla de normalización, podemos re-procesar desde Bronze sin llamar otra vez a las APIs externas. Bronze es el event log inmutable; todo lo demás es una proyección derivable.
Este patrón se conoce como Medallion Architecture, popularizado por Databricks. Es el estándar actual para data platforms a escala.
04 · Fuentes de datos

Los proveedores del hub

Así como un centro logístico recibe de proveedores por carretera, barco, avión y tren, la plataforma acepta datos por cuatro canales distintos:

Canal 1

APIs REST (HTTP)

Servicios web que entregan JSON bajo demanda. Típicamente registros públicos, regulaciones, fuentes de terceros con tarifas.

Canal 2

Archivos CSV / SFTP

Exports batch de sistemas ERP o bancarios, entregados por sftp o drop en bucket. Volúmenes grandes, formato tabular.

Canal 3

Bases de datos externas

Replicación incremental desde sistemas legacy usando watermark-based extraction: solo se trae lo nuevo o modificado desde el último run.

Canal 4

Parquet en S3

Archivos columnares comprimidos. Eficientes para procesamiento analítico. Son el output natural de otros sistemas big-data upstream.

Contrato común: todos los extractores producen el mismo tipo de objeto (ExtractBatch) con metadata de lineage. Downstream no necesita saber ni le importa de qué canal vino el dato.
05 · Extractors

Los equipos de ingesta

Cada canal tiene un operador especializado. Todos implementan el mismo contrato (async def extract() -> AsyncIterator[ExtractBatch]) pero saben manejar las particularidades de su fuente.

RestAPIExtractor

Maneja paginación (offset/cursor/page), reintentos exponenciales con tenacity, respeto a rate limits vía Retry-After, autenticación por headers. Emite métricas por request.

CSVExtractor

Usa Polars scan_csv en modo lazy. Lee en chunks configurables para mantener la memoria acotada aunque el archivo sea de varios GB.

PostgresExtractor

Incremental basado en watermark. Guarda el último updated_at procesado en tabla dedicada. Re-ejecutar es idempotente.

S3ParquetExtractor

Itera Parquet por row-groups (no carga el archivo entero). Soporta column pruning: solo lee las columnas que se necesitan downstream.

Lineage tagging: cada batch lleva batch_id, source_system, extracted_at. Esa metadata viaja con el dato hasta el destino final. Si en 6 meses aparece un registro anómalo en producción, se puede rastrear exactamente al archivo/API origen.
06 · Primera línea de defensa

Validación estructural con Pydantic

En el muelle de recepción, cada caja se abre y se inspecciona antes de entrar al almacén. La primera línea de defensa es Pydantic v2, que valida la forma básica de cada registro:

Tipos y presencia

¿El campo amount es un Decimal? ¿document_date es una fecha real?

Rangos y tamaños

Una moneda ISO 4217 tiene exactamente 3 letras. El risk score va de 0 a 100.

Expresiones regulares

¿El hash tiene formato SHA-256? ¿El código postal cumple el patrón de su país?

Coherencia cross-field

¿end_date es posterior o igual a start_date? ¿total_value >= 0?

Pydantic también normaliza: "usd", "USD", " usd " llegan todos como USD. Un espacio extra en el nombre de una empresa se elimina. La semántica canónica vive en un solo lugar.
Performance: Pydantic v2 corre sobre un core escrito en Rust. Valida alrededor de 1 millón de modelos por segundo. No es el cuello de botella del pipeline.
07 · Segunda línea de defensa

Reglas de negocio como código (YAML)

Pydantic cubre la forma. Las reglas de negocio viven en archivos YAML versionados en git, ejecutados por un motor propio sobre DataFrames de Polars:

suite: transactions_silver
rules:
  - name: transaction_reference_unique
    type: unique
    column: reference
    severity: error          # Bloquea el pipeline

  - name: amount_within_range
    type: range
    column: amount
    params: { min: -1000000000, max: 1000000000 }
    severity: error

  - name: transaction_type_known
    type: in_set
    column: transaction_type
    params:
      values: ["INVOICE", "PAYMENT", "REFUND", "ADJUSTMENT", "FEE"]
    severity: warning        # Solo emite métrica
Ventaja para el equipo: un analista de negocio puede proponer una regla nueva en un PR sin escribir Python. El historial queda en git y las revisiones son trazables.
Tipos soportados: not_null, unique, regex, in_set, range, row_count_range, expression (Polars arbitrario).
08 · Tercera línea de defensa

Validación estadística con Great Expectations

Algunas reglas no son binarias. Por ejemplo: "el monto promedio de transacciones no debería variar más del 20% día a día". Para eso usamos Great Expectations:

Detección de drift

Si la distribución cambia bruscamente respecto al baseline, alerta. Útil para detectar cambios silenciosos upstream.

Expectations estadísticas

Media, mediana, cuantiles, KL divergence. Todo dentro de rangos esperados.

Data Docs HTML

Genera documentación estática auto-explicativa que se puede compartir con stakeholders no-técnicos.

Las tres líneas de defensa son complementarias:
LíneaVelocidadFocoEjecución
1. PydanticMuy rápidaForma, tiposIn-pipeline, por registro
2. YAML rulesRápidaReglas de negocioIn-pipeline, por batch
3. Great ExpectationsModeradaPropiedades estadísticasPost-load o scheduled
09 · Transformación

De Bronze a Silver: la zona de preparación

En el centro logístico, entre el muelle y el almacén hay una zona donde los productos se etiquetan, empaquetan y se asignan a slots. En datos, los transformers hacen lo equivalente:

Normalizer

Aplica las reglas de canonización: mayúsculas consistentes, whitespace, formatos de fecha ISO, monedas ISO 4217. Todo lo que garantiza que "ACME Corp" y "acme corp" sean el mismo registro downstream.

SCD2 Handler (Slowly Changing Dimension Type 2)

Cuando un atributo de una entidad cambia (ej: el risk score de un cliente), en vez de sobrescribir, se inserta una nueva versión con valid_from / valid_to / is_current. Habilita consultas point-in-time: "¿cuál era el risk score de CP-123 el 2024-06-01?"

Enricher

Agrega campos derivados: risk tiers (LOW/MODERATE/HIGH/CRITICAL desde un score numérico), hashes de fila para detección de cambios, etiquetas geográficas por country_code.

¿Por qué SCD2 es importante para compliance? Si un auditor pregunta "¿qué información tenía la empresa sobre este cliente el día que le aprobamos crédito?", SCD2 responde en un solo query. Sin SCD2 habría que reconstruir desde logs.
10 · Loaders

Los montacargas y bandas transportadoras

Cuando el producto está listo, hay que depositarlo en su slot del almacén. Los loaders son los responsables de escribir al destino final con la máxima eficiencia posible:

PostgresBulkLoader

Usa el protocolo COPY nativo vía asyncpg. En lugar de INSERT uno-por-uno, streamea datos como un único flujo continuo.

~1Krows/s INSERT 150K+rows/s COPY

S3ParquetLoader

Escribe Parquet particionado por ingestion_date. Consumidores downstream (Athena, Redshift Spectrum, Spark) hacen predicate pushdown y solo leen las particiones relevantes.

Patrón UPSERT via staging:
  1. Crear tabla temporal con la misma estructura que la tabla destino.
  2. TRUNCATE la staging.
  3. COPY los nuevos datos a staging.
  4. INSERT ... ON CONFLICT (...) DO UPDATE desde staging hacia destino.
Resultado: la velocidad de COPY con la semántica de UPSERT idempotente.
11 · Source of Truth

PostgreSQL: el almacén maestro

El corazón operativo del sistema. Todos los datos que representan el estado actual del negocio viven aquí, con garantías ACID y constraints de integridad:

legal_entities

Empresas y personas en documentos legales

legal_documents

Sentencias, regulaciones, actas. Particionada por mes.

counterparties

Clientes, proveedores, partners

contracts

Contratos comerciales con vigencias

transactions

Facturas, pagos, ajustes. Particionada por mes.

regulations

Normativas con rangos de vigencia

dim_counterparty

Dimensión SCD2 con histórico

stg_*

Tablas UNLOGGED para staging rápido

¿Por qué PostgreSQL y no un data warehouse como Redshift o Snowflake? Porque necesitamos transacciones ACID para operaciones críticas (contratos, facturación), constraints de integridad (FK, CHECK) y joins relacionales complejos. Para analytics puro, Gold puede exportarse a Redshift/Athena.
12 · Optimización · Particionado

Pasillos por zona, no un solo galpón

Una tabla de transacciones crece ~1M rows/día en escala media. A los 2 años tendrías 700M rows en una sola tabla. Sin particionar, todo query sufre.

Solución: particionado nativo por RANGE(fecha) con granularidad mensual.

transactions_2024_01 [30M rows] transactions_2024_02 [30M rows] transactions_2024_03 [30M rows] ... transactions_2026_04 [30M rows] transactions_default (catch-all para fechas fuera de rango)

Query: "último mes"

Sin particiones: full index scan sobre 700M → ~800ms

Con particiones: partition pruning + scan sobre 30M → ~40ms

Retention: borrar > 5 años

Sin particiones: DELETE masivo, horas, WAL enorme

Con particiones: DROP PARTITION → O(1)

Mantenimiento: una función plpgsql create_monthly_partition(parent, date) pre-crea los próximos 12 meses. En producción se automatiza con pg_partman o un cron mensual.
13 · Optimización · Índices

El sistema de localización interno

Un almacén grande tiene un sistema de coordenadas (pasillo, estante, nivel). Los índices de Postgres cumplen la misma función: decirle al planner dónde están los datos sin escanear todo.

Usamos varios tipos, cada uno con un caso de uso justificado:

BTree compuesto

(counterparty_id, transaction_date DESC) — soporta queries "últimas N transacciones de un cliente" con index scan puro.

BRIN

Para columnas time-ordered. ~10 KB por millón de rows vs ~30 MB de un BTree. Aprovecha que los inserts son cronológicos.

GIN + pg_trgm

Búsqueda fuzzy: legal_name ILIKE '%ACME%' con performance sub-lineal.

Partial index

WHERE status = 'ACTIVE' — el índice solo cubre ~30% de las filas, reduciendo tamaño y overhead de escritura.

Covering (INCLUDE)

INCLUDE (amount, currency) habilita index-only scans: el planner responde sin visitar el heap.

GIN jsonb_path_ops

Para queries sobre columnas JSONB. 3× más rápido que el operador default en queries de contención @>.

Trade-off: cada índice tiene un costo de escritura. Se agregan solo cuando hay un query real que los justifica (medido con pg_stat_statements).
14 · Optimización · Materialized Views

Reportes pre-computados

Algunas preguntas del negocio son caras de resolver desde cero:

"Total facturado por contraparte, por mes, por moneda, de los últimos 3 años"

Computar ese agregado ad-hoc sobre 700M transacciones puede tardar minutos. En vez de eso, mantenemos una vista materializada precomputada que se refresca en horarios valle:

CREATE MATERIALIZED VIEW mv_monthly_revenue_per_counterparty AS
SELECT
    c.external_id, c.name, c.country_code,
    date_trunc('month', t.transaction_date)::DATE AS month,
    t.currency, COUNT(*), SUM(t.amount), AVG(t.amount)
FROM transactions t JOIN counterparties c ON c.id = t.counterparty_id
WHERE t.transaction_type IN ('INVOICE', 'PAYMENT', 'REFUND')
GROUP BY 1, 2, 3, 4, 5;

-- Índice único requerido para refresh CONCURRENTLY
CREATE UNIQUE INDEX ux_mv_revenue_key
    ON mv_monthly_revenue_per_counterparty (external_id, month, currency);
REFRESH CONCURRENTLY: el refresh corre en background sin bloquear las lecturas. Los consumidores siguen viendo la versión anterior hasta que el swap es atómico. Sin esto, refrescar una MView grande cuelga la DB por minutos.
15 · Optimización · Connection pooling

PgBouncer: multiplexación de conexiones

Cada conexión a PostgreSQL consume ~10 MB de RAM en el backend. Con 500 conexiones simultáneas eso son 5 GB solo en conexiones vacías. La mayoría del tiempo, las apps no están ejecutando queries.

Sin pooling: Con PgBouncer (transaction mode): App1 ─┐ App1 ─┐ App2 ─┤ 500 conexiones App2 ─┤ ┌─ Backend1 App3 ─┤ directas al backend App3 ─┤ ├─ Backend2 ... ├─→ 500 procesos PG ... ├─→ ├─ Backend... App500┘ App500┘ └─ Backend25 (25 conexiones)

Ratio típico

500:25 = 20:1 de apps a backend connections.

Modo transaction

Una conexión backend se reasigna entre clientes entre transacciones. Máxima eficiencia.

Restricciones del transaction mode con asyncpg: hay que deshabilitar el statement cache (statement_cache_size=0) porque PgBouncer rota backend connections. Operaciones con session state (LISTEN/NOTIFY, SET LOCAL) requieren session mode o el direct engine.
16 · AWS · Storage

S3: el almacén externo de capacidad ilimitada

Mientras PostgreSQL es el almacén operativo, S3 es el archivo histórico y el data lake. Tres propiedades lo hacen indispensable:

Durabilidad

99.999999999% (11 nueves). Estadísticamente, si guardas 10 millones de objetos, perderías uno cada 10 mil años.

Costo

~$0.023/GB/mes en STANDARD, ~$0.004/GB/mes en GLACIER_IR. Lifecycle policies automatizan la degradación.

Integración

Athena, Glue, Redshift Spectrum, EMR, Lambda — todo el ecosistema AWS consume S3 nativamente.

Organizamos S3 en cuatro buckets (uno por capa del Medallion):

bronze

Raw inmutable

silver

Clean, normalizado

gold

Agregados

quarantine

Rechazos

Desarrollo local: usamos MinIO, un clon open-source de S3 que corre en un contenedor Docker. API idéntica, cero costo, cero dependencia de credenciales AWS durante dev.
17 · AWS · Event-driven

Lambda: compute reactivo a eventos

Cuando un archivo aterriza en Bronze, no queremos que nadie lo descubra manualmente. El sistema debe reaccionar automáticamente:

S3 ObjectCreated event ↓ Lambda "s3-event-ingest" (construye payload → envía a SQS) ↓ SQS queue "ldp-ingest" ↓ Lambda "sqs-pipeline-trigger" (POST a Prefect API) ↓ Prefect dispara el flow en un worker ECS
¿Por qué no ejecutar el pipeline en Lambda directamente?
  • Timeout máximo: 15 minutos. Procesar batches grandes puede tomar más.
  • Memoria máxima: 10 GB. DataFrames grandes no caben.
  • Deploy package: 50 MB zipped / 250 MB unzipped. Pandas + dependencies excede.
Lambda solo dispara; el procesamiento pesado corre en ECS Fargate o workers Prefect dedicados.
Serverless: Lambda factura por invocación y GB-segundo. Si no hay archivos, el costo es cero. Ideal para workloads bursty y event-driven.
18 · AWS · Messaging

SQS: desacoplamiento y control de concurrencia

Entre el trigger S3 y el pipeline colocamos una cola SQS como buffer. Esto resuelve tres problemas:

Picos de carga

Si llegan 1000 archivos de golpe, SQS los serializa. El pipeline los procesa a su ritmo.

Reintentos automáticos

Si el consumer falla, SQS redelivers el mensaje hasta maxReceiveCount veces (5 por defecto).

Dead Letter Queue

Mensajes que fallan persistentemente van a DLQ. Una alarma CloudWatch dispara cuando llegan.

Producer (S3 event) → SQS ldp-ingest → Consumer (Lambda) │ │ (5 intentos fallidos) ↓ SQS ldp-ingest-dlq │ ↓ CloudWatch Alarm → PagerDuty / Slack
Partial batch response: configuramos Lambda con ReportBatchItemFailures. Si un batch de 10 mensajes tiene 1 que falla, solo ese 1 se re-enqueue. Los otros 9 no se reprocesan.
19 · AWS · Big data

AWS Glue: procesamiento distribuido

El pipeline Python + Polars maneja cómodamente hasta ~100 millones de rows en un solo proceso. Cuando los volúmenes superan eso (por ejemplo, un backfill histórico de 3 años), necesitamos distribuir el trabajo.

Python + Polars (single-process): Glue PySpark (distributed): 1 worker, 16 GB RAM N workers en paralelo Hasta ~100M rows cómodo Hasta ~10B rows cómodo Latencia: minutos Latencia: horas (para volúmenes masivos) Costo: ~$0.05/hora (Fargate) Costo: ~$0.44/DPU-hora

Glue se encarga del aprovisionamiento, la distribución del trabajo vía Spark y la persistencia en el Glue Data Catalog para que Athena y Redshift Spectrum descubran el schema automáticamente.

Regla práctica:
  • < 100M rows por batch → Python + Polars (más simple, más barato)
  • 100M–10B rows → Glue PySpark
  • > 10B rows continuos → considerar EMR managed o arquitectura lakehouse (Iceberg, Delta)
20 · Orquestación

Prefect: el sistema de gestión del almacén

Un centro logístico sin WMS (Warehouse Management System) es caos. Prefect cumple ese rol para los datos: define qué corre, cuándo, con qué dependencias, con qué retries y con qué observabilidad.

Flows y Tasks

Un @flow es un pipeline completo. Un @task es cada unidad atómica (extract, normalize, load). Prefect construye el DAG dinámicamente en runtime.

Retries y backoff

@task(retries=3, retry_delay_seconds=30) — retrying automático con backoff exponencial configurable.

Schedules

Cron, interval, rrule. Un deployment asocia un flow a un schedule y un work pool (local, Docker, K8s, ECS).

UI y observabilidad

Timeline de ejecución, logs por task, gráfico de dependencias, estados de runs. Todo en localhost:4200 (dev) o Prefect Cloud (prod).

¿Por qué Prefect y no Airflow? Prefect tiene DX superior (flows en Python puro sin DSL), mejor modelo de deployments y observability nativa. Para equipos de 3–15 ingenieros es más productivo. Airflow sigue ganando a escala 500+ DAGs con ecosystem maduro de providers.
21 · Infraestructura como código

Terraform: los planos versionados

Aprovisionar S3 + SQS + Lambda + Glue + IAM roles + KMS + Aurora en la consola web de AWS es lento, no auditable y no reproducible. Terraform resuelve los tres problemas:

Declarativo

Describes el estado deseado. Terraform calcula el diff y aplica solo los cambios necesarios.

Versionado

Todos los recursos viven en git. git blame te dice quién cambió el security group y cuándo.

Reproducible

Crear un entorno dev idéntico a prod: terraform workspace new dev + apply. Minutos, no días.

Portable

Multi-cloud (AWS, GCP, Azure, Cloudflare). El mismo tooling para toda la infra.

Estructura del módulo: main.tf (provider), s3.tf (buckets + KMS + lifecycle), sqs.tf (queue + DLQ + alarm), lambda.tf (funciones + IAM), glue.tf (catalog + jobs), rds.tf (Aurora Serverless v2), variables.tf, outputs.tf.
State backend: el archivo terraform.tfstate vive en un bucket S3 con versioning + DynamoDB para locking. Así varios ingenieros pueden colaborar sin pisarse.

23 · Observabilidad

Los tres pilares: logs, métricas, tracing

Sin observabilidad no hay operación profesional. Tres componentes complementarios:

structlog (logs)

Emisión en JSON estructurado. Cada log incluye pipeline, run_id, módulo, función, línea. Agregable en CloudWatch/Datadog/ELK sin regex. Correlación por run_id a través de todas las stages.

{"event":"batch_loaded","pipeline":"legal_ingestion",
 "run_id":"20260424T021500","rows":4950,"duration_ms":820}

Prometheus (métricas)

Counters y histogramas siguiendo el método RED + data-specific:

  • ldp_records_extracted_total{source, pipeline}
  • ldp_records_loaded_total{target, pipeline}
  • ldp_records_quarantined_total{pipeline, rule}
  • ldp_pipeline_duration_seconds{pipeline, stage}
  • ldp_dq_checks_total{suite, result}

Grafana (visualización + alerting)

Dashboards construidos sobre Prometheus. Alertas: tasa de rechazo > 5% por 10 min, DLQ > 0, p95 de duración del stage load > 60s.

Troubleshooting

Problemas comunes y soluciones

"port 5432 already in use"

Otro PostgreSQL local está corriendo. Detenlo (pg_ctl stop, service manager, o Docker Desktop) o cambia el puerto en docker-compose.yml a 5433.

"connection refused" hacia postgres

El healthcheck de Postgres tarda unos segundos. Espera 10s después de docker compose up. Verifica con docker compose logs postgres que aparezca "database system is ready to accept connections".

"ModuleNotFoundError: No module named 'src'"

Falta instalar en modo editable. Ejecuta pip install -e ".[dev]" desde la raíz del proyecto.

Pipeline rechaza el 100% de las filas

Muy probable que falten migraciones o el seed. Corre en orden: alembic upgrade head → python scripts/seed_data.py → make pipeline.

"InvalidAccessKeyId" contra S3 local

MinIO no arrancó correctamente o el .env tiene credenciales distintas a las del docker-compose.yml. Verifica ambos.

Prefect UI no carga

Revisa docker compose logs prefect. Si dice "port 4200 in use", detén el proceso anterior. Si dice "waiting for database", espera 30s adicionales.
